Existing Lease Onboarding Wizard — Approaches

Two architectural approaches for the batch wizard. Both use the same linear-per-tenant + table-overview UX.

A

Dedicated Bulk Endpoint (Recommended)

New backend endpoint: POST /v1/properties/{id}/leases:bulk-onboard

Accepts an array of tenant+lease objects. Creates Tenants, TenantAccounts, and Leases in a single transaction. Skips the TenantApplication flow entirely.

Pros

  • Single API call — atomic success/failure
  • No invoice/payment/signing overhead
  • Clean separation from the application flow
  • Can validate all units are available upfront

Cons

  • New endpoint + service method to build
  • Large payload for many tenants
B

Reuse Existing Endpoints

Frontend orchestration: Call existing create-tenant-application + approve endpoints sequentially for each tenant.

The wizard collects data, then submits each tenant through the existing pipeline with flags to skip invoice/payment requirements.

Pros

  • Less backend work — reuse existing services
  • Each tenant processed independently

Cons

  • Multiple API calls — partial failure risk
  • Must hack around invoice/payment/signing requirements
  • Slower — sequential API calls per tenant
  • Pollutes TenantApplication table with synthetic records
