# Changelog

## 2026-07-17 — Global tenants list (real) + property filter
- `TenantsScreen` (`/more/tenants`) is now real — replaced the fully mocked static-seed-data screen. New `TenantApi`/`TenantModel`/`TenantsNotifier` (paginated, search + status filter, mirrors `BlocksNotifier`/`UnitsNotifier`).
- Discovered the backend endpoint already existed and needed no new work: `GET /api/v1/admin/clients/{client_id}/tenants` (`ListTenantsAcrossProperties`, shipped in #374 but never consumed by any frontend — confirmed via `git show --stat` that the commit was backend-only). Supports a repeatable `property_id` filter, `status` (`ACTIVE`/`EXPIRED` — has vs. doesn't have an active lease), search (`query`+`search_fields=first_name,last_name,phone`), and standard pagination. Access-scoped server-side (`ResolvePropertyScopeFilter` — OWNER sees everything, ADMIN/STAFF only assigned properties; a disallowed `property_id` 403s the whole request rather than being silently dropped).
- Response nests a `recent_lease` (status, rent, dates, and a `unit` object) per tenant — the `unit` shape is byte-for-byte identical to the existing `UnitModel` (backend uses the same `DBAdminUnitToRest` transform for both), so `TenantLeaseRef.unit` reuses `UnitModel` directly instead of a parallel type.
- **One tenants page, multiple entry points** (not a separate property-scoped screen): the property detail page's "Tenants" Manage-grid tile now pushes `/more/tenants?property_id=...&property_name=...`; `TenantsScreen` shows a removable "Property: {name}" filter chip when arriving this way — exact same `?block_id=...` pattern already used for units-by-block, just a different field name and no dedicated screen since this one already covers "list tenants, optionally filtered."
- **Dropped from the mock**: the "balance owed" figure the old static data showed — this endpoint has no balance field (that lives in invoices/expenses, a separate #374 endpoint not integrated here). Tenant detail (`/more/tenants/:id`) stays mocked for now.
- Modules affected: `api/tenant_api.dart` (new), `repository/models/tenant_model.dart` (new), `repository/notifiers/tenants/tenants_notifier.dart` (new), `modules/main/tenants/root.dart` (rewritten), `modules/main/properties/detail.dart`, `navigation/routes.dart`.

## 2026-07-17 — Block delete restrictions + info icon
- Block creation and deletion are now both gated to `property.type == 'MULTI'` (previously only the entry points into the create flow were gated — the `BlocksListScreen` FAB and the delete action itself weren't). Viewing/editing blocks stays available on SINGLE properties.
- Deleting a block on a MULTI property now additionally requires `unitsCount == 0` — the "⋯" sheet's Delete row is disabled with an inline reason ("This block has units — move or delete them first" / "Single-unit properties can't delete blocks") rather than opening a confirm dialog that would just fail. This goes beyond what the web UI restricts (confirmed it has no such guard) — an explicit ask, not a web-parity decision like the rest of the blocks work.
- Added an info icon to the Blocks list header (`RLBackHeader.trailing`) opening a short "What are blocks?" explainer dialog.

## 2026-07-17 — Property blocks CRUD + units-by-block filter
- Property blocks are now fully CRUD from mobile (previously read-only, used only for the add-unit block picker and the Manage grid count). Confirmed via the web app's block modules first: no block detail page exists there (flat list → edit directly, or a "⋯" menu → Edit/Delete), block creation is gated to `property.type == 'MULTI'` (backend 404s otherwise, same as units), and deleting a block is **not** restricted by existing units (unlike deleting an occupied unit) — mirrored all three decisions as-is rather than inventing new mobile-only behavior.
- New `BlocksListScreen` (`properties/blocks_list.dart`, `/properties/:id/blocks`) — search + status filter chips + pagination (mirrors `units_list.dart`), rendered as cards (image, name, status pill, units count, created date, description snippet) rather than plain rows since blocks carry more info than a unit row. FAB → Add. Tapping a card opens Edit directly (no detail page, matches web); a "⋯" icon opens a sheet with Edit/Delete. Reachable from the property detail page's now-tappable **Blocks** Manage-grid tile.
- New `AddBlockScreen`/`EditBlockScreen` (`properties/add_block.dart`, `properties/edit_block.dart`) — Name (required), single-image upload (`UImagePicker` gained an optional `maxImages` param for this — web restricts block photos to one, unlike units' bulk upload), Description (optional), Status (Active/Maintenance/Inactive, default Active). Real `POST`/`PATCH` via new `CreateBlockNotifier`/`UpdateBlockNotifier`; delete via new `DeleteBlockNotifier` (real `DELETE`, plain confirm dialog, no unit-count guard).
- **"View Units" per block**: `UnitsQuery` gained a `blockId` field, `UnitApi.getUnits()` now sends `block_ids` (backend already supported it, just wasn't wired), and `units_list.dart` accepts `blockId`/`blockName` — passed via GoRouter query params (`?block_id=...&block_name=...`, a new pattern in this app's routing) from each block card's "View Units" button. Shows a removable "Block: {name}" filter chip above the status filters when active.
- Data layer: `PropertyBlockApi` gained `getBlock`/`createBlock`/`updateBlock`/`deleteBlock`; `PropertyBlockModel` extended with `description`/`images`/`createdAt`/`updatedAt`; new `BlocksNotifier` (paginated list state, mirrors `UnitsNotifier`) and `blockDetailProvider` (single-block family fetch, mirrors `unitDetailProvider`) — distinct from the existing `propertyBlocksProvider` (a flat, ungated, pageSize-100 fetch still used for the add-unit block picker and the Manage grid count).
- Modules affected: `api/property_block_api.dart`, `repository/models/property_block_model.dart`, `repository/notifiers/blocks/{blocks,create_block,update_block,delete_block}_notifier.dart` (new), `repository/providers/properties/block_detail_provider.dart` (new), `modules/main/properties/{blocks_list,add_block,edit_block}.dart` (new), `modules/main/properties/{detail,unit_form_widgets,units_list}.dart`, `api/unit_api.dart`, `repository/notifiers/units/units_notifier.dart`, `navigation/routes.dart`.

## 2026-07-17 — Real image upload on Add/Edit Unit
- Replaced the mocked, non-functional image placeholder on the Add/Edit Unit forms with a real, working multi-image picker + uploader (`UImagePicker` in `unit_form_widgets.dart`) — pm_mobile's first working image upload anywhere in the app.
- Discovered this didn't need new backend work: `apps/go` (the tenant Flutter app) already uploads through the property-manager web app's R2 proxy (`https://pm.rentloopapp.com/api/r2/upload`, unauthenticated by design — trusts `objectKey`+`file` blindly, already shipped in two other apps). New `R2UploadService` (`api/r2_upload_service.dart`) ports that exact implementation: `objectKey` prefixed `property/images/` (same bucket path convention the web add/edit forms use), returns the resulting public URL.
- `UImagePicker` mirrors `apps/go`'s `ManageAttachmentsWidget`: a thumbnail grid + "Add" tile (camera or photo library, via a new `image_picker` dependency), each picked image uploads immediately with a per-tile progress spinner / retry-on-failure / remove button, and a tap-to-preview full-screen swipeable viewer that handles both freshly-picked local files and already-uploaded (or, in Edit, prefilled) network images. Reports the uploaded URL list and an "any upload still in flight" flag up to the parent form; both Add and Edit block submission (with a validation message, not a disabled-looking button) while any photo is still uploading — matches the web form's `uploadingIds.length > 0` guard.
- `UnitApi.updateUnit()`/`UpdateUnitNotifier` gained an `images` param (wasn't there before — Edit previously had no way to touch images at all).
- New native permissions: iOS `NSCameraUsageDescription`/`NSPhotoLibraryUsageDescription`, Android `CAMERA`/`READ_EXTERNAL_STORAGE`/`WRITE_EXTERNAL_STORAGE` (copied from `apps/go`, reworded for unit photos).
- Modules affected: `api/r2_upload_service.dart` (new), `api/unit_api.dart`, `repository/notifiers/units/update_unit_notifier.dart`, `modules/main/properties/unit_form_widgets.dart`, `modules/main/properties/{add_unit,edit_unit}.dart`, `constants.dart` (`kR2UploadUrl`), `pubspec.yaml` (`image_picker`), `ios/Runner/Info.plist`, `android/app/src/main/AndroidManifest.xml`.

## 2026-07-17 — Unit edit/delete
- Added a settings gear icon to `unit_detail.dart`'s header (mirrors the property detail page), pushing a new `UnitSettingsHubScreen` (`properties/unit_settings.dart`, `/properties/:id/units/:unitId/settings`) — same visual pattern as `PropertySettingsHubScreen`: identity card, an "Edit" tile, and (only when `property.type == 'MULTI'`) a "Delete unit" tile in a danger-zone section, disabled with an explanatory subtitle when the unit is Occupied/PartiallyOccupied.
- Added a real, wired `EditUnitScreen` (`properties/edit_unit.dart`, `/properties/:id/units/:unitId/edit`), ported field-for-field from the web edit-unit page: same fields as Add minus Block and Status (neither is editable there either), prefilled from the current unit. Rental Information (occupancy/rent/billing cycle) is replaced with an info banner and omitted from the PATCH body when the unit is Occupied/PartiallyOccupied — matches the web's `isRentalInfoEditable` guard and the backend's own restriction. Submits via a new `UpdateUnitNotifier` to the real `PATCH .../units/{unitId}` endpoint.
- Delete wired to the real `DELETE .../units/{unitId}` endpoint via a new `DeleteUnitNotifier`, behind a confirm `AlertDialog` (mirrors `PropertySettingsHubScreen`'s archive-confirmation style). On success, invalidates the property's stats/units-preview/blocks providers and refreshes the units list, then `context.go()`s straight to the units list (skips popping back through the now-deleted unit's detail screen). Added an `'UnitIsOccupied'` case to `translateApiErrorMessage()` for the server-side 403 this can't fully be prevented client-side against (e.g. a unit gets occupied in another session mid-flow).
- Extended `UnitModel` with `maxOccupantsAllowed` (needed to prefill the edit form's occupancy stepper — wasn't previously read anywhere).
- Refactored: `add_unit.dart`'s field widgets (type/status rows, text inputs, chips, occupants stepper, bottom action bar) moved to a new shared `unit_form_widgets.dart` (public `U`-prefixed versions) since `edit_unit.dart` needed ~80% of the same UI — avoids a second ~400-line duplicate.
- Modules affected: `api/unit_api.dart` (`updateUnit()`, `deleteUnit()`), `repository/models/unit_model.dart`, `repository/notifiers/units/{update_unit,delete_unit}_notifier.dart` (new), `modules/main/properties/unit_form_widgets.dart` (new), `modules/main/properties/{edit_unit,unit_settings}.dart` (new), `modules/main/properties/{add_unit,unit_detail}.dart`, `lib/api_error_messages.dart`, `navigation/routes.dart`.
