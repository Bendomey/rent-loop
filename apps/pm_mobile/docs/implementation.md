# Implementation

## Entry Points
- `lib/main.dart` — `SystemChrome` overlay style, `runApp(ProviderScope(child: RentloopManagerApp()))`
- `lib/src/app.dart` — `RentloopManagerApp`; builds `MaterialApp.router` with GoRouter (`buildRoutes(ref)`) and `buildTheme()`
- `lib/src/navigation/routes.dart` — `buildRoutes(ref)`: defines all `GoRoute`/`StatefulShellRoute` entries + auth redirect logic

## Per-Module Breakdown

### api/
- **Entry point:** `lib/src/api/root.dart` — `AbstractApi` base class
- **Key classes:** `AbstractApi` (`execute()`: GET/POST/PATCH/DELETE, attaches `Authorization: Bearer <token>` unless `authRequired: false`, throws `ApiException` on ≥400), `ApiException` (parses `json['errors']['message']`, `null`-safe on malformed bodies)
- `user_api.dart` — `UserApi extends AbstractApi`: `login({email, password})` (unauthenticated, `POST /api/v1/admin/users/login`, returns `UserLoginResult{token, user}`), `getMe()` (authenticated, `GET /api/v1/admin/users/me`, returns `UserModel`)
- `property_api.dart` — `PropertyApi extends AbstractApi`: `getProperties({clientId, page, pageSize, ids, order, orderBy, status, search})` (`GET /api/v1/admin/clients/{client_id}/properties`, returns `PropertiesPage{rows: List<PropertyModel>, meta: PaginationMetaModel}`); `status` is the full dotted enum value (e.g. `Property.Status.Active`), `search` maps to `query`+`search_fields=name`. Also `getProperty({clientId, propertyId})` (`GET /api/v1/admin/clients/{client_id}/properties/{property_id}`, returns a single `PropertyModel`)
- `unit_api.dart` — `UnitApi extends AbstractApi`: `getUnits({clientId, propertyId, page, pageSize, status, search, blockId})` (`GET /api/v1/admin/clients/{client_id}/properties/{property_id}/units`, `status` = dotted `Unit.Status.*` value, `search` maps to `query`+`search_fields=name`, `blockId` maps to `block_ids` (backend takes an array; sent as a one-element list) — same shape as `PropertyApi.getProperties()`; returns `UnitsPage{rows: List<UnitModel>, meta: PaginationMetaModel}`); `getUnit({clientId, propertyId, unitId})` (`GET .../units/{unitId}?populate=Property,PropertyBlock`); `createUnit({clientId, propertyId, blockId, name, description, images, type, status, rentFee, rentFeeCurrency, paymentFrequency, maxOccupantsAllowed})` (`POST .../properties/{property_id}/blocks/{block_id}/units`, real — backs the Add Unit form); `updateUnit({clientId, propertyId, unitId, name?, description?, images?, type?, area?, rentFee?, rentFeeCurrency?, paymentFrequency?, maxOccupantsAllowed?})` (`PATCH .../units/{unitId}` — every param optional and only sent when non-null, so a caller can omit fields it didn't show an editor for, e.g. rental info on an occupied unit); `deleteUnit({clientId, propertyId, unitId})` (`DELETE .../units/{unitId}`, no body, 204 — the backend 403s with `UnitIsOccupied` if the unit is currently occupied)
- `property_block_api.dart` — `PropertyBlockApi extends AbstractApi`: `getBlocks({clientId, propertyId, page, pageSize, status, search})` (`GET /api/v1/admin/clients/{client_id}/properties/{property_id}/blocks`, `pageSize` defaults to 100 for the picker/count callers — the paginated list screen passes 10; `status`/`search` same shape as units'); `getBlock({clientId, propertyId, blockId})` (`GET .../blocks/{block_id}`); `createBlock({clientId, propertyId, name, description, images, status})` (`POST .../properties/{property_id}/blocks`, real — backs Add Block); `updateBlock({clientId, propertyId, blockId, name?, description?, images?, status?})` (`PATCH .../blocks/{block_id}` — same optional-fields-only shape as `updateUnit()`); `deleteBlock({clientId, propertyId, blockId})` (`DELETE .../blocks/{block_id}`, no body, 204 — no occupancy-style restriction, confirmed the web UI doesn't have one either)
- `analytics_api.dart` — `AnalyticsApi extends AbstractApi`: `getToken({clientId})` (`GET /api/v1/admin/clients/{client_id}/analytics/token`, returns a short-lived Cube JWT); `CubeApi` (plain class, not `AbstractApi` — different host/token): `load({token, query})` POSTs to `$kCubeApiUrl/cubejs-api/v1/load`, returns the raw `data` rows
- `tenant_api.dart` — `TenantApi extends AbstractApi`: `getTenants({clientId, page, pageSize, status, search, propertyId})` — the first list endpoint in this app not nested under a property in the URL (`GET /api/v1/admin/clients/{client_id}/tenants`, handler `ListTenantsAcrossProperties`, spans every property the caller can access); `status` is the plain value `ACTIVE`/`EXPIRED` (not a dotted enum, unlike units/properties/blocks), `search` maps to `query`+`search_fields=first_name,last_name,phone`, `propertyId` maps to the backend's repeatable `property_id` param (sent as a one-element list, same shape as `UnitApi.getUnits()`'s `blockId`→`block_ids`) — access-scoped server-side via `ResolvePropertyScopeFilter`, an inaccessible id 403s the whole request. Returns `TenantsPage{rows: List<TenantModel>, meta: PaginationMetaModel}`

### architecture/
- **Entry point:** `lib/src/architecture/app_startup/app_startup_notifier.dart` (replaced the old hand-written mock at `architecture/app_startup.dart`, now deleted)
- **Key class:** `AppStartupNotifier` — `@Riverpod(keepAlive: true)` (converted from the hand-written `Notifier`, now matching the documented `CLAUDE.md`/`apps/go` convention)
- **States:** `AppStartupStatus { loading, unauthenticated, workspaceSelect, ready, error }` — unchanged shape from the mocked version; the extra `workspaceSelect` step vs. `apps/go` remains PM-app-specific (landlords pick a workspace/organization after login, tenants don't)
- **Methods, all real:**
  - `init()` — checks connectivity → reads JWT via `TokenManager` (null → `unauthenticated`) → `GET /me`; an `ApiException` (dead/expired token) drops to `unauthenticated` silently rather than showing `error`, which is reserved for connectivity/unexpected failures
  - `completeLogin(UserModel user)` — called by `LoginNotifier` after it has already saved the JWT; runs the same session-entry logic as `init()`
  - `selectWorkspace(ClientUserModel)` — called from the workspace picker; persists + sets the workspace via `CurrentWorkspaceNotifier.select()`, then `ready`
  - `logout()` — clears JWT (`TokenManager.remove()`), workspace id (`WorkspaceIdManager.remove()`), both notifiers, back to `unauthenticated`
  - `_enterSession(UserModel user)` (private) — shared by `init()`/`completeLogin()`: sets `CurrentUserNotifier`, reads the persisted workspace id, calls `resolveWorkspace()`; auto-selects and goes `ready` if resolved, else `workspaceSelect`
- `current_user/current_user_notifier.dart` — `CurrentUserNotifier`, keepAlive, state `UserModel?`, `setUser()`/`clear()`
- `current_workspace/current_workspace_notifier.dart` — `CurrentWorkspaceNotifier`, keepAlive, state `ClientUserModel?`, `select(ClientUserModel)` (persists via `WorkspaceIdManager` then sets state) / `clear()`
- `secure_storage/`, `token_manager/`, `workspace_id_manager/` — thin `@riverpod` provider wrappers around the plain classes in `lib/`

### lib/
- `storage.dart` — abstract `Storage` interface (`read`/`delete`/`write`)
- `secure_storage.dart` — `SecureStorage extends Storage`, wraps `FlutterSecureStorage`
- `token_manager.dart` — `TokenManager`: `save()`/`get()`/`remove()` JWT under key `rentloop_manager.token`
- `workspace_id_manager.dart` — `WorkspaceIdManager`: same shape, key `rentloop_manager.current_client_id`
- `workspace_resolution.dart` — `isActiveClientUser(ClientUserModel)` (last-dot-segment case-insensitive match against `"active"`, e.g. `"ClientUser.Status.Active"` → `"active"` — deliberately not a substring `contains` check, which would misclassify `"Inactive"`), `resolveWorkspace(clientUsers, {storedClientId})` (auto-selects on exactly one active membership, or a stored id matching an active one; else `null`)
- `api_error_messages.dart` — `translateApiErrorMessage({errorMessage, defaultErrorMessage})`; cases so far: `PasswordIncorrect`/`UserNotFound` (login), `UnitIsOccupied` (delete-unit's 403 when the unit is currently occupied — client-side already disables the delete tile for this case, but the server check is authoritative against races), default otherwise — mirrors `apps/go`'s pattern of adding specific cases as they're discovered
- `property_status.dart` — `propertyStatusLabel(status)` (last-dot-segment, mirrors `workspace_resolution.dart`'s convention), `propertyTypeLabel(type)` (`SINGLE`/`MULTI` → display label)
- `money.dart` — `pesewasToCedis(int pesewas)` (mirrors `apps/go`'s `MoneyLib.pesawasToCedis` convention), `cedisToPesewas(num cedis)` (inverse — converts a form-entered cedis amount back to the pesewas int the backend expects)
- `unit_status.dart` — `unitTypeLabel(type)` (APARTMENT/HOUSE/STUDIO/OFFICE/RETAIL → display label), `shouldShowSeeAllUnits(totalUnits)` (the property detail screen's units preview caps at 5 — this is `total >= 6`)
- `property_stats_logic.dart` — `parseCubeNum(value)` (Cube returns every measure as a string), `currentMonthDateRange(now)` (ISO date-range helper for the Cube Invoices query), `computePropertyStats({unitsRows, invoiceRows, leaseRows, bookingRows, applicationRows})` (combines 5 separate Cube query results into one `PropertyStats`)

### navigation/
- `routes.dart` — `buildRoutes(ref)`: internal `_RouterNotifier` (`ChangeNotifier`) listens to `appStartupNotifierProvider` (renamed from the old `appStartupProvider`) and calls `notifyListeners()` on change, wired as GoRouter's `refreshListenable`. Redirect `switch` on `AppStartupStatus`: `loading`/`error` → `/splash`, `unauthenticated` → `/auth/welcome`, `workspaceSelect` → `/auth/workspace-select`, `ready` → `/` (main shell)
- Global `GoRouter? appRouter` — reserved for notification-driven navigation from outside the widget tree (mirrors `apps/go`'s pattern), not yet used since no push notifications are wired up
- `splash.dart` — `SplashScreen`; calls `appStartupNotifierProvider.notifier.init()` via `addPostFrameCallback`

### modules/auth/
- `welcome/root.dart` → `login/root.dart` → `workspace_select/root.dart`
- `login/root.dart` — email/password form; submits via `loginNotifierProvider.notifier.submit()`. Loading/error state comes from `loginNotifierProvider` (not local state), so the router's redirect never fires mid-request. The error banner's dismiss handler clears both a local validation error and, via `LoginNotifier.reset()`, any API-sourced failure.
- `workspace_select/root.dart` — reads `client_users` off `currentUserNotifierProvider` (already fetched at login/`/me`, no extra API call). Active memberships (`isActiveClientUser`) are tappable and call `AppStartupNotifier.selectWorkspace()`; non-active ones render dimmed/disabled with a status pill. Role pill tone: `OWNER`/`MANAGER` (case-insensitive) → danger, else neutral.

### repository/
- `api_state.dart` — `ApiStatus { idle, pending, success, failed }` (with `isLoading()`/`isSuccess()`/`isFailed()`), `ApiState { status, errorMessage }`
- `models/user_model.dart`, `client_user_model.dart`, `client_model.dart` — `@JsonSerializable` DTOs matching the confirmed live API schema (`UserModel.clientUsers` defaults to `[]` when the `client_users` key is absent, not throw)
- `notifiers/auth/login_notifier.dart` — `LoginNotifier` (`LoginState extends ApiState`): `submit({email, password})` runs `UserApi.login()` → `TokenManager.save()` → `AppStartupNotifier.completeLogin()` → success, all in one try/catch (`ApiException` → translated message, other errors → default message); `reset()` returns to a clean idle state, called only from the login screen's error-dismiss handler
- `notifiers/properties/properties_notifier.dart` — `PropertiesNotifier` (`PropertiesState{items, total, hasNextPage, currentPage, isLoadingMore, isLoading, error}`): `loadFirstPage(PropertiesQuery{search, status})` resets to page 1, `loadNextPage()` appends page N+1 (pageSize 10) — the first paginated-list notifier in this app, modeled on `apps/go`'s `MaintenanceRequestsNotifier`
- `notifiers/units/units_notifier.dart` — `UnitsNotifier` (`UnitsState{items, total, hasNextPage, currentPage, isLoadingMore, isLoading, error}`): `loadFirstPage(propertyId, [UnitsQuery query])` / `loadNextPage()` — `UnitsQuery{search, status}` mirrors `PropertiesQuery`, same shape as `PropertiesNotifier`
- `notifiers/units/create_unit_notifier.dart` — `CreateUnitNotifier` (`CreateUnitState extends ApiState`): `submit({propertyId, blockId, name, description, images, type, status, rentFee, rentFeeCurrency, paymentFrequency, maxOccupantsAllowed})` runs `UnitApi.createUnit()` in a try/catch (same shape as `LoginNotifier`); `reset()` clears a previous API-sourced failure — backs `add_unit.dart`
- `notifiers/units/update_unit_notifier.dart`, `delete_unit_notifier.dart` — `UpdateUnitNotifier`/`DeleteUnitNotifier` (same `ApiState` shape): `submit()` wraps `UnitApi.updateUnit()`/`deleteUnit()`; back `edit_unit.dart`/`unit_settings.dart` respectively
- `notifiers/blocks/blocks_notifier.dart` — `BlocksNotifier` (`BlocksState{items, total, hasNextPage, currentPage, isLoadingMore, isLoading, error}`): `loadFirstPage(propertyId, [BlocksQuery query])` / `loadNextPage()` — `BlocksQuery{search, status}`, exact same shape as `UnitsNotifier`/`UnitsQuery`. Distinct from `propertyBlocksProvider` (see below) — this one backs the paginated `blocks_list.dart` screen; that one is a flat pageSize-100 fetch for the add-unit picker and Manage grid count.
- `notifiers/blocks/create_block_notifier.dart`, `update_block_notifier.dart`, `delete_block_notifier.dart` — `CreateBlockNotifier`/`UpdateBlockNotifier`/`DeleteBlockNotifier` (same `ApiState` shape as the unit notifiers): wrap `PropertyBlockApi.createBlock()`/`updateBlock()`/`deleteBlock()`; back `add_block.dart`/`edit_block.dart`/`blocks_list.dart`'s delete-confirm flow respectively
- `notifiers/tenants/tenants_notifier.dart` — `TenantsNotifier` (`TenantsState{items, total, hasNextPage, currentPage, isLoadingMore, isLoading, error}`): `loadFirstPage([TenantsQuery query])` / `loadNextPage()` — same paginated shape as `UnitsNotifier`/`BlocksNotifier`, except `loadFirstPage` takes no required `propertyId` (unlike Units/Blocks, which are always constructed scoped to a property); `TenantsQuery{search, status, propertyId}` treats property as just one optional filter among several, since `TenantApi.getTenants()` itself is cross-property by design. Backs the global `tenants/root.dart` list, with or without a pre-seeded `propertyId`
- `providers/properties/property_detail_provider.dart`, `property_stats_provider.dart`, `property_units_preview_provider.dart`, `property_blocks_provider.dart` — `@riverpod Future<T>` family providers keyed by property id: real property data, Cube-derived `PropertyStats`, a 5-item units preview page, and a property's blocks page (`pageSize` 100), respectively
- `providers/properties/unit_detail_provider.dart` — `unitDetailProvider(propertyId, unitId)`, a two-key `@riverpod Future<UnitModel>` family provider; calls `UnitApi.getUnit()` with `populate=Property,PropertyBlock` (mirrors the web unit detail page's populate call)
- `providers/properties/block_detail_provider.dart` — `blockDetailProvider(propertyId, blockId)`, same two-key family shape as `unitDetailProvider`; calls `PropertyBlockApi.getBlock()` — backs `edit_block.dart`'s prefill (a dedicated single-fetch, not a client-side lookup into the cached `propertyBlocksProvider` list, so it's correct even if that list hasn't loaded yet)
- `models/unit_model.dart`, `property_stats_model.dart`, `property_block_model.dart` — `UnitModel` (`@JsonSerializable`, subset of `AdminOutputUnit`: id/name/type/status/description/rentFee/rentFeeCurrency/paymentFrequency/maxOccupantsAllowed/area/images/propertyId/property/propertyBlockId/propertyBlock/createdAt/updatedAt), `UnitPropertyRef`/`UnitBlockRef` (minimal id+name nested refs), `PropertyStats` (plain class, aggregate of 5 Cube queries, not a single API response — `occupancyPercent` getter), `PropertyBlockModel` (id/name/description/images/status/unitsCount/createdAt/updatedAt)
- `models/tenant_model.dart` — `TenantModel` (id/firstName/lastName/email/phone/profilePhotoUrl/recentLease, `fullName` getter), `TenantLeaseRef` (status/unit/rentFee/rentFeeCurrency/paymentFrequency/moveInDate/moveOutDate — nested on `recent_lease`, active-preferred-else-most-recent per the backend). `TenantLeaseRef.unit` reuses `UnitModel` directly rather than a duplicate type, since the backend's `recent_lease.unit` JSON is produced by the same `DBAdminUnitToRest` transform used everywhere else `UnitModel` is populated

### modules/main/
Five `StatefulShellRoute` tab branches — `home`, `properties`, `activity`, `money`, `more` — each with a `root.dart` plus feature-specific add/detail/settings files. `tenants/` and `announcements/` are reachable via nested routes off `more/` and the main shell respectively.

Real (wired to `currentUserNotifierProvider`/`currentWorkspaceNotifierProvider`):
- `home/root.dart` — `_TopHeader`'s workspace-name eyebrow and manager avatar. The rest of the screen (revenue card, stat grid, needs-attention, collection trend, quick actions) is still mocked.
- `more/root.dart` — `_ProfileCard` (name/email) and `_WorkspaceCard` (workspace name/role — the previously-fabricated `properties · units` stat was dropped, since that data isn't in the API response). Logout button calls `appStartupNotifierProvider.notifier.logout()`.
- `workspace_sheet.dart` — the in-shell workspace switcher (`showWorkspaceSheet(context)`, opened from Home's eyebrow and More's workspace card — note the signature dropped its old `activeId` param, since the sheet now reads the active workspace off the provider itself). Lists `client_users`; tapping an active one calls `currentWorkspaceNotifierProvider.notifier.select()`; inactive ones are dimmed/non-tappable, same convention as `workspace_select/root.dart`.
- `properties/root.dart` — the list screen is real: `propertiesNotifierProvider`-backed infinite scroll (10/page), pull-to-refresh, debounced search, status filter chips.
- `properties/detail.dart` — the detail screen is real: four independent providers (property/stats/units-preview/blocks) each with their own skeleton/error handling; swipeable hero image carousel (falls back to a type-colored placeholder tile when a property has no images); Manage grid is 6 tiles in 2 rows of 3 — **Blocks** (real count via `propertyBlocksProvider`, tappable → `blocks_list.dart`), **Units**, Tenants, Leases, Applications, Bookings (Cube-sourced, Bookings only when the property's `modes` includes `BOOKING`); units preview caps at 5 with a "See all" action (hidden below 6 total units) that pushes `units_list.dart`. Unit rows push `unit_detail.dart`. The "Units" section label shows an add-icon (via `RLLabel.trailing`) that pushes `add_unit.dart` when `property.type == 'MULTI'`.
- `properties/units_list.dart` — paginated "all units" screen for a property, `UnitsNotifier`-backed infinite scroll (10/page) with debounced search and a status filter chip row (All/Available/Occupied/Maintenance/Draft) — mirrors `root.dart`'s (properties list) search+filter+pagination pattern. Unit rows push `unit_detail.dart`. Shows a FAB and an empty-state CTA, both pushing `add_unit.dart`, when `property.type == 'MULTI'` (watches `propertyDetailProvider` for the type check). Accepts optional `blockId`/`blockName` (via GoRouter query params `?block_id=...&block_name=...`, set by a block card's "View Units" button in `blocks_list.dart`) — shows a removable "Block: {name}" filter chip above the status filters when present, clearing it re-fetches unfiltered.
- `properties/unit_detail.dart` — real single-unit page (`unitDetailProvider`-backed), ported from the web unit detail view: hero image/placeholder (same visual pattern as `detail.dart`'s property hero), status+type pills, a Rent Fee card, and an `RLFilterChips`-styled tab strip (Details/Leases/Maintenance Requests/Tenant Reviews). Only Details renders real content (Description, Property & Block, Specifications, Timeline cards, using the label/value `_FieldRow` pattern from `tenants/detail.dart`); the other three tabs show an inline "coming soon" placeholder since Leases/Maintenance/Reviews have no real mobile data source yet. Header carries a settings gear (mirrors `detail.dart`'s) pushing `unit_settings.dart`. No status-change mutation yet (the dropdown web has on its unit page) — everything else (create/edit/delete) is real.
- `properties/unit_form_widgets.dart` — shared field widgets/constants for the unit create/edit forms, also reused (for the simpler fields) by the block create/edit forms below (public `U`-prefixed versions, since Dart's leading-underscore privacy is file-scoped): `kUnitTypes`, `kRentFeeLabels`, `kBillingCycleLabels`, `USectionHeading`, `USelectRow`, `UFieldLabel`, `UTextInput`, `UImagePicker`, `UChip`, `UOccupantsStepper`, `UBottomActionBar` (takes a `submitLabel` so Add/Edit can each show their own button text). `UImagePicker` is real (not mocked) — see the Image upload entry below; it also takes an optional `maxImages` (null = unlimited, used by units; blocks pass `1` since the web form caps block photos at one).
- `properties/add_unit.dart` — real, wired "Add Unit" form (`/properties/:id/units/add`), single scrollable form (not a step wizard) ported field-for-field from the web create-unit page: block (required, bottom-sheet picker backed by `propertyBlocksProvider` — a property with zero blocks shows an in-app "Add a block" CTA pushing `add_block.dart`, the picker refreshes via provider invalidation on return), unit type (5 rows), status (Draft/Available/Maintenance — Add-only, kept local to this file since Edit doesn't touch status), name, real photo upload (`UImagePicker`), description, occupancy (single/shared tenant-count stepper), currency (GHS only)+rent fee, billing cycle (Daily always, Weekly/Monthly only when the property's `modes` includes `LEASE`). Submits via `CreateUnitNotifier` to the real create-unit endpoint (rejects submit with a validation message, not a silently-disabled button, if any photo is still uploading); on success invalidates `propertyStatsProvider`/`propertyUnitsPreviewProvider`/`propertyBlocksProvider`, refreshes `unitsNotifierProvider`, then pops. Dropped from v1 (present in the web form but omitted here): Area, Features.
- `properties/edit_unit.dart` — real, wired "Edit Unit" form (`/properties/:id/units/:unitId/edit`), same fields as Add minus Block and Status (web doesn't expose either for editing), prefilled from `unitDetailProvider` on first load (`_prefilled` guard flag, set directly during `build()` rather than via `initState`/hooks, since the unit data arrives async) — `UImagePicker` is seeded with `initialImages: unit.images`, shown as already-uploaded and never re-uploaded. Rental Information (occupancy/currency+rent/billing cycle) is replaced with an `RLInlineBanner` and the corresponding fields are omitted from the `UpdateUnitNotifier.submit()` call — not just hidden — when the unit's status is Occupied or PartiallyOccupied, matching both the web form's `isRentalInfoEditable` guard and the backend's own restriction on those fields. On success invalidates `unitDetailProvider`/`propertyUnitsPreviewProvider`, refreshes `unitsNotifierProvider`, then pops.
- **Image upload (real)** — `api/r2_upload_service.dart`'s `R2UploadService.uploadFile(File)`: `http.MultipartRequest` POSTs `file`+`objectKey` (`property/images/{timestamp}-{random-hex}.{ext}`) to `kR2UploadUrl` (`constants.dart`), returns the resulting public URL. This is a Cloudflare R2 proxy hosted by the **property-manager web app**, not `services/main` — same endpoint `apps/go` (tenant app) already uploads maintenance-request photos through, unauthenticated by design (trusts `objectKey`/`file` blindly). `UImagePicker` drives it: `image_picker` for camera/gallery, per-image `UImageStatus{idle,uploading,done,failed}` state machine, immediate upload on pick, tap-to-preview full-screen swipeable viewer (`Image.file` for freshly-picked/unsent, `Image.network` for already-uploaded/prefilled).
- `properties/unit_settings.dart` — `UnitSettingsHubScreen`, same identity-card-plus-tile-list pattern as `settings/hub.dart` (`PropertySettingsHubScreen`): an identity card (name, type · status), an "Edit" tile pushing `edit_unit.dart`, and — only when `property.type == 'MULTI'` — a "Delete unit" tile in a danger-zone section, disabled (greyed, no chevron, no tap) with an explanatory subtitle when the unit is Occupied/PartiallyOccupied. Delete opens a confirm `AlertDialog` (mirrors `PropertySettingsHubScreen._showArchiveAlert`'s style) wired to `DeleteUnitNotifier`; on success invalidates the same provider cluster as Add/Edit and calls `context.go('/properties/:id/units')` — a `go()`, not a `pop()`, so it doesn't land back on the now-deleted unit's detail screen.
- `properties/blocks_list.dart` — real, wired "Blocks" list (`BlocksNotifier`-backed infinite scroll, 10/page, debounced search + status filter chips — All/Active/Maintenance/Inactive), reached from the property detail page's Manage-grid Blocks tile. Renders cards (image or `Icons.dashboard_outlined` placeholder, name, status pill, "{n} units", created date, description snippet) rather than plain rows — mirrors `_SingleUnitCard`'s "bigger card, more info" visual choice from `detail.dart` rather than `units_list.dart`'s plain `RLRow` list, since block cards carry more info. No block detail page (confirmed against web) — tapping a card pushes `edit_block.dart` directly; a "⋯" icon opens a bottom sheet (Edit/Delete). Each card's "View Units" button pushes `units_list.dart` pre-filtered to that block. FAB → `add_block.dart`.
- `properties/add_block.dart`, `edit_block.dart` — real, wired Add/Edit Block forms, same structural pattern as `add_unit.dart`/`edit_unit.dart` (scroll-to-top-on-error, validation, `UBottomActionBar`) but far fewer fields: Name (required), single-image `UImagePicker` (optional, `maxImages: 1`), Description (optional, textarea), Status (Active/Maintenance/Inactive row-selector, default Active — no Occupied-style restriction on any field, blocks don't have that concept). Edit prefills from `blockDetailProvider`. Submit via `CreateBlockNotifier`/`UpdateBlockNotifier`; on success invalidates `propertyBlocksProvider`/`propertyStatsProvider` (Add) or `propertyBlocksProvider`/`blockDetailProvider` (Edit) and refreshes `blocksNotifierProvider`, then pops.
- `add.dart` (property add) /`edit_sheets.dart`/`settings/` remain mocked.
- `tenants/root.dart` — `TenantsScreen(propertyId?, propertyName?)`, real, `TenantsNotifier`-backed global tenants list (every property the caller can access), reached off `more/`. One screen serves two entry points rather than two separate screens: a plain `/more/tenants` push (unfiltered) and `properties/detail.dart`'s Manage-grid "Tenants" tile pushing `/more/tenants?property_id=...&property_name=...` (both params optional, read by `routes.dart` off the query string) — mirrors `units_list.dart`'s block-filter pattern exactly, down to the removable "Property: {name}" filter chip and its clear handler. Search bar (debounced, `search_fields=first_name,last_name,phone`) + status filter chips (All/Active/Expired, mapped to the plain `ACTIVE`/`EXPIRED` values, not a dotted enum) + paginated `RLRow`-based list (infinite scroll, pull-to-refresh). Each row: `_TenantAvatar` (profile photo or `RLAvatar` initials fallback), title `tenant.fullName`, subtitle `unit.name · property.name` off `recentLease` (falls back to `tenant.phone` when there's no lease), trailing `RLPill` with the lease status (reuses `propertyStatusLabel`/`statusTone`) or a neutral "No lease" pill. ## Non-obvious Logic
- `AppStartupStatus.workspaceSelect` is a PM-app-specific state with no `apps/go` equivalent — landlords may belong to multiple organizations/workspaces and must pick one after login, before the JWT session is considered "ready"
- The login screen intentionally avoids writing to `appStartupNotifierProvider` state during the in-flight request (loading/error live in `loginNotifierProvider` instead), to prevent the router's `redirect` callback from bouncing the user to `/splash` while the request is pending
- `resolveWorkspace()`'s auto-select rule: exactly one active membership always wins outright (ignoring any stored id); otherwise a stored id is only honored if it still points at an *active* membership — a stale id pointing at a now-deactivated membership is silently ignored, not auto-selected
- `isActiveClientUser()` matches on the last dot-segment of the raw status string, not a substring `contains` — this was a real bug caught in review (`"Inactive"` contains the substring `"active"`, so a naive `contains('active')` check would misclassify a deactivated membership as active)
- `docs/` is checked into git (not gitignored) for this app, matching the convention already established in `apps/go/docs/`
- `propertyStatusLabel()` (in `property_status.dart`) is reused as-is for unit statuses too (`Unit.Status.*` dot-splits the same way `Property.Status.*` does) — no separate `unitStatusLabel()` was written
- The property detail screen's stats card and Manage grid share one `propertyStatsProvider` fetch (5 parallel Cube queries) rather than each cell issuing its own query — Cube.js has no single multi-cube query in the REST `/load` endpoint used here, so each measure group (Units, Invoices, Leases, Bookings, TenantApplications) is still a separate `CubeApi.load()` call, just run concurrently via `Future.wait` and parsed together by `computePropertyStats()`
