# Architecture

## Project Type
Flutter / Dart 3.8.1+ mobile app (property-manager / landlord-facing client for RentLoop rental platform). Companion to the React property-manager portal (`apps/property-manager`). Targets iOS and Android. **Auth is real** (login, cold-start token validation, workspace selection, logout, against `https://api.rentloopapp.com`) as of the 2026-07-11 login integration — see `docs/decisions.md` and `docs/superpowers/specs/2026-07-10-login-integration-design.md`. **The properties list is also real** (paginated, 10/page, infinite scroll, search + status filter) as of the 2026-07-16 properties list integration — see `docs/superpowers/specs/2026-07-16-properties-list-integration-design.md`. **The property detail page is also real** (property data, Cube-sourced stats card + Manage grid, paginated units list) as of the 2026-07-16 property detail integration — see `docs/superpowers/specs/2026-07-16-property-detail-integration-design.md`. **The global tenants and leases lists are also real** (see the Tenants/Leases sections below). Every other module/screen (properties add/settings, tenant detail, lease detail, activity, money, announcements) is still UI-only against mock/static data, pending its own integration pass.

## Data Flow
**Auth (real):**
1. `SplashScreen` calls `appStartupNotifierProvider.notifier.init()` on first frame
2. `AppStartupNotifier.init()` checks connectivity → reads JWT via `TokenManager` → if present, calls `GET /api/v1/admin/users/me`; an `ApiException` (dead/expired token) drops to `unauthenticated` silently, any other failure sets `error`
3. `LoginScreen` submits via `LoginNotifier.submit()` → `UserApi.login()` (`POST /api/v1/admin/users/login`) → saves JWT → calls `AppStartupNotifier.completeLogin(user)`
4. Both cold-start and post-login funnel through `AppStartupNotifier._enterSession()`: sets `CurrentUserNotifier`, then runs `resolveWorkspace()` — auto-selects if exactly one active `client_user` membership, or if a previously-persisted workspace id (via `WorkspaceIdManager`) still matches an active membership; otherwise status → `workspaceSelect` and `WorkspaceSelectScreen` shows the full membership list (inactive ones disabled)
5. Selecting a workspace calls `AppStartupNotifier.selectWorkspace()` → `CurrentWorkspaceNotifier.select()` (persists + sets state) → status `ready` → `GoRouter` (via `refreshListenable`) redirects into the main `StatefulShellRoute` (5 tabs: home, properties, activity, money, more)
6. Logout clears JWT, workspace id, and both notifiers, back to `unauthenticated`

**Property detail (real):** `PropertyDetailScreen` watches four independent family providers keyed by property id — `propertyDetailProvider` (`GET .../properties/{id}`), `propertyStatsProvider` (Cube-sourced: occupancy/revenue/leases/bookings/applications, via `AnalyticsApi.getToken` + `CubeApi.load`), `propertyUnitsPreviewProvider` (first 5 units), and `propertyBlocksProvider` (used both for the Manage grid's Blocks tile count and the add-unit block picker). Each section shows its own shimmer skeleton / inline error independently — one failing doesn't block the others. "See all" under the units preview (shown only when total units ≥ 6) pushes `/properties/:id/units`, a `UnitsNotifier`-backed infinite-scroll list with debounced search and a status filter (All/Available/Occupied/Maintenance/Draft) — same `UnitsQuery`/search+filter pattern as the properties list. Unit rows on both the preview and the full list push `/properties/:id/units/:unitId`, a `unitDetailProvider`-backed single-unit page (hero image, status/type pills, Rent Fee card, tabbed Details/Leases/Maintenance Requests/Tenant Reviews — only Details has real content) whose header carries a settings gear into `/properties/:id/units/:unitId/settings`.

**Add Unit (real, MULTI properties only):** `/properties/:id/units/add` (`AddUnitScreen`) is a single scrollable form — not a step wizard — ported field-for-field from the web create-unit page: block (required, picked from a bottom sheet backed by `propertyBlocksProvider`; a property with zero blocks shows an in-app "Add a block" CTA pushing `add_block.dart`, refreshing the picker via provider invalidation on return), unit type, status (Draft/Available/Maintenance), name, real photo upload, description, occupancy (single/shared tenant count), currency+rent fee, billing cycle (options depend on the property's `modes`). Submits via `CreateUnitNotifier` (`ApiState` mutation pattern, same shape as `LoginNotifier`) to the real `POST .../blocks/{blockId}/units` endpoint; on success it invalidates `propertyStatsProvider`/`propertyUnitsPreviewProvider`/`propertyBlocksProvider` and refreshes `unitsNotifierProvider` before popping back. Entry points (FAB on the units list, CTA in its empty state, icon next to the property detail page's "Units" section label) are all gated on `property.type == 'MULTI'`, mirroring the backend's own 404 on SINGLE properties. Its field UI (`USelectRow`, `UFieldLabel`, `UTextInput`, `UChip`, `UOccupantsStepper`, `UImagePicker`, `UBottomActionBar`, etc.) lives in `unit_form_widgets.dart`, shared with `edit_unit.dart` and (for the simpler fields) `add_block.dart`/`edit_block.dart`.

**Property Blocks (real CRUD):** `/properties/:id/blocks` (`BlocksListScreen`) — search + status filter chips + pagination via a new `BlocksNotifier` (mirrors `UnitsNotifier`), rendered as cards (image, name, status pill, units count, created date, description) since blocks carry more info than a unit row. No block detail page exists (confirmed against the web app first) — a card tap opens `EditBlockScreen` directly (viewing/editing works on any property type), a "⋯" icon opens a sheet with Edit/Delete. Add/Edit (`add_block.dart`/`edit_block.dart`): Name, single-image upload (`UImagePicker` with `maxImages: 1` — web caps block photos at one, unlike units' bulk upload), Description, Status (Active/Maintenance/Inactive). **Create and delete are both gated to `property.type == 'MULTI'`** — the Add FAB is hidden entirely on SINGLE properties, and the sheet's Delete row is disabled (with an inline reason) rather than opening a confirm dialog that would just fail. Delete additionally requires `unitsCount == 0` even on MULTI properties — an explicit ask that goes beyond what the web UI restricts (web has no such guard). A header info icon opens a short "What are blocks?" explainer dialog. Reachable from the property detail page's Manage-grid "Blocks" tile (now tappable). Each block card's "View Units" button pushes the units list pre-filtered to that block via GoRouter query params (`?block_id=...&block_name=...`) — `UnitsQuery` gained a `blockId` field and `units_list.dart` shows a removable "Block: {name}" chip when active. `propertyBlocksProvider` (flat, pageSize-100, ungated) stays as-is for the add-unit block picker and the Manage grid count; the new `BlocksNotifier`/`blockDetailProvider` are separate, purpose-built for the paginated list and single-block edit fetch respectively.

**Image upload (real):** `UImagePicker` (in `unit_form_widgets.dart`, used by both Add and Edit) uploads directly to Cloudflare R2 via `R2UploadService` → `kR2UploadUrl`, which is **not** a `services/main` endpoint — it's the property-manager *web app's* own R2 proxy route (`pm.rentloopapp.com/api/r2/upload`, unauthenticated by design), the same one `apps/go` (the tenant app) already uploads maintenance-request photos through. No new backend work was needed; this just ports an already-shipped pattern to a third app. Each picked image (camera or library, via `image_picker`) uploads immediately with its own progress/retry/remove state; Edit's picker starts pre-seeded from the unit's existing `images` (shown as already-uploaded, never re-uploaded). Both forms block submit — with a validation message, not silently — while any photo is still mid-upload.

**Edit/Delete Unit (real):** `/properties/:id/units/:unitId/settings` (`UnitSettingsHubScreen`) is the same identity-card-plus-tile-list pattern as `PropertySettingsHubScreen` — an "Edit" tile pushes `/properties/:id/units/:unitId/edit` (`EditUnitScreen`, same fields as Add minus Block/Status, prefilled, `UpdateUnitNotifier` → real `PATCH`); a "Delete unit" tile (only when `property.type == 'MULTI'`, disabled when the unit is Occupied/PartiallyOccupied) opens a confirm dialog wired to `DeleteUnitNotifier` → real `DELETE`. Both mutations invalidate the same cluster of providers as Add (`propertyStatsProvider`/`propertyUnitsPreviewProvider`/`propertyBlocksProvider`/`unitsNotifierProvider`); delete additionally `context.go()`s straight to the units list rather than popping back through the now-deleted unit's detail screen. Edit hides Rental Information (occupancy/rent/billing) behind an info banner and omits those fields from the PATCH body when the unit is Occupied/PartiallyOccupied, mirroring both the web form and a backend-enforced restriction.

**Tenants (real, list only):** `/more/tenants` (`TenantsScreen`) is **one global list, not property-scoped by construction** — unlike every other list in this app, `GET .../clients/{client_id}/tenants` isn't nested under a property; it spans every property the caller can access, with an optional repeatable `property_id` query param to narrow it (access-checked server-side — a disallowed id 403s rather than being dropped). `TenantsNotifier`/`TenantsQuery` mirror `BlocksNotifier` exactly, `search_fields=first_name,last_name,phone`, `status` is `ACTIVE`/`EXPIRED` (has vs. lacks an active lease) rather than a dotted enum. The property detail page's "Tenants" Manage-grid tile pushes this same screen with `?property_id=...&property_name=...` pre-applied (same query-param-filter mechanism as the units-by-block filter) — there's deliberately no separate "this property's tenants" screen. Each row's `recent_lease.unit` reuses `UnitModel` directly (identical JSON shape, same backend transform). No balance/payment figure — that's a separate, unintegrated invoices endpoint. Tenant detail (`/more/tenants/:id`) is still mocked.

**Leases (real, list only):** `/more/leases` (`LeasesScreen`) mirrors `TenantsScreen`'s "one global list, entry-point-agnostic" design exactly — `GET .../clients/{client_id}/leases` (handler `ListLeasesAcrossProperties`, added in the same #374 backend PR as the tenants endpoint and likewise unconsumed by any frontend until now) spans every property the caller can access, narrowed by an optional repeatable `property_id`. Unlike tenants, `status` here is the **full dotted `Lease.Status.*` enum** (`Pending`/`Active`/`Completed`/`Cancelled`/`Terminated` — matches the web property-scoped leases table's status filter exactly, confirmed against `apps/property-manager/app/modules/properties/property/occupancy/leases/`), not a simplified value — so the mobile status chips map directly to `Lease.Status.$label` rather than going through a translation function. Search maps to `query`+`search_fields=code` (the web table searches by lease code too). The request always sends `populate=Unit,Tenant` so each row's nested `unit`/`tenant` are populated — this is the standard, always-honored `populate` query param (independent of the property-scoped preload gate that caused the tenants `recent_lease` bug above), so no backend changes were needed here. `unit`/`tenant` reuse `UnitModel`/`TenantModel` directly (same transforms, and `DBAdminTenantToRest` — used for a lease's nested tenant — has no `recent_lease` key, so there's no circular reference back to `LeaseModel`). The property detail page's "Leases" Manage-grid tile pushes this same screen with `?property_id=...&property_name=...` pre-applied, identical to the Tenants tile. Row tap pushes `/more/leases/:id`, which renders `RLComingSoon` — no lease detail page exists yet (mobile or otherwise) to build against.

**Every other module** (activity, money, announcements, most of `more/`, and tenant/lease *detail*) still renders local mock data — no API calls in those trees yet. The pattern is proven and ready to replicate: `AbstractApi.execute()` → resource `XxxApi` class → Riverpod query provider / `ApiState` mutation notifier → screen (see `docs/patterns.md`), one module at a time.
